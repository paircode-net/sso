## Objetivo

Entregar uma plataforma centralizada de **Single Sign-On (SSO)** baseada em **OpenIddict**, com identidade em **ASP.NET Identity**, padrões **OAuth 2.1**, **OpenID Connect** e **JWT**, cobrindo autenticação, autorização e gestão de identidade para todos os produtos do ecossistema.

A solução deve permitir:

- identidade única compartilhada entre produtos;
- autorização híbrida (Roles + Claims + permissões dinâmicas);
- multi-produto, multi-organização (tenant) e multi-filial (branch);
- contexto completo de autorização: **Usuário → Organização → Branch → Produto → Role → Claims → Permissões**.

## Escopo

### Inclui (visão do produto — feature 00001)

- Cadastro e gerenciamento de usuários
- Autenticação centralizada (OAuth 2.1 / OIDC)
- Gerenciamento de credenciais
- Recuperação e redefinição de senha
- Confirmação de e-mail
- Autenticação em dois fatores (2FA)
- Gerenciamento de sessões
- Emissão e revogação de tokens
- Auditoria de autenticação e autorização
- Roles e Claims globais e por produto
- Permissões customizadas e dinâmicas (sem recompilação das apps cliente)
- Políticas de autorização (herança de permissões via roles; **sem** herança Branch pai→filho no MVP — D4)
- Multi-produto (AuthClients OAuth, scopes, roles/claims/menus por produto)
- Multi-organization (tenants, isolamento lógico, IdPs externos por org)
- Multi-branch (filiais, vínculo usuário×branch, roles/claims por branch ativa)
- Resolução de autorização no contexto completo da requisição
- UI login/consent Razor em `SSO.Web.Api` (MVP — D6)
- Admin MVP via API apenas (D11)

### Fora de escopo (fase inicial / explícito)

- Remoção do aggregate Sample antes da Fase 2 estável
- Portal admin rico (UI além de login/consent)
- Implementação simultânea de todos os IdPs — ordem: Entra ID → Google → LDAP (D7)
- Microserviços / event bus distribuído
- Observabilidade avançada (APM) e CI/CD — P-002 / P-003
- Compliance formal além do playbook de security
- Herança automática de permissões Branch pai → filho (MVP)

### Premissa de entrega

A feature **00001** é **épico**. Implementação em **fases sequenciais**. Cada fase: PR revisável, testes e atualização de `CONTEXT/`.

## Abordagem

### Princípios de arquitetura

1. Preservar **Clean Architecture + CQRS Full** (Sample como espelho de estrutura).
2. Composition root em `SSO.Middleware`; AuthN/AuthZ habilitados de forma completa (não parcial).
3. Domain não depende de OpenIddict/Identity — portas no Domain; adapters em Infrastructure (ADR-001, ADR-002).
4. Writer/Reader + `CommitAsync` como unidade de trabalho.
5. Rotas de gestão: `api/identity/{resource}`; protocolo OIDC em `/connect/*`.
6. Login/consent: Razor Pages/Views no host `SSO.Web.Api` (D6).
7. Classes `sealed`, Ids `Guid`, localization `pt-BR` (+ `en-US` nos testes).
8. Soft-delete + auditoria nas entidades Identity (D12 / P-008 parcialmente fechado para Identity).

### Bounded context (decidido — ADR-006)

| Context | Schema / DbContext | Uso |
|---------|--------------------|-----|
| Default | `DefaultDb` / `DefaultDbContext` | Sample (scaffold) |
| **Identity** | `IdentityDb` / `IdentityDbContext` | Domínio SSO + stores Identity/OpenIddict |

### Modelo de domínio (aggregates / entidades raiz previstas)

```text
Identity (context)
├── Organization          # tenant
├── Branch                # filial (hierarquia estrutural opcional; sem herança authz MVP)
├── Product               # produto do ecossistema (≠ AuthClient)
├── User                  # extensão Identity (perfil / credenciais)
├── Membership            # User ↔ Organization [↔ Branch]
├── Role                  # domínio SSO: global | por Product | por Branch
├── ClaimDefinition       # claim tipada (global/produto)
├── Permission            # permissão dinâmica
├── RolePermission        # vínculo role → permission
├── UserRoleAssignment    # role no contexto Org/Branch/Product
├── UserClaimAssignment   # claim no contexto Org/Branch/Product
├── ProductEnablement     # Organization → Products habilitados
├── AuthClient            # cliente OAuth (por Product)
├── Scope                 # escopos OIDC/OAuth
├── Session               # sessões ativas (ou view sobre tokens/persisted grants)
├── ExternalIdentityProvider
└── AuthAuditEvent        # auditoria AuthN/AuthZ
```

OpenIddict entities no mesmo `IdentityDbContext` (store EF) — ADR-001.

### Contexto de autorização (runtime) — ADR-003, ADR-004, ADR-005

```text
Login OIDC → token → switch-context → novo token (organization_id/branch_id + permissions efetivas)
  → APIs/products autorizam via claims do JWT
```

| Dimensão | Origem |
|----------|--------|
| User | `sub` do token |
| Organization | claim `organization_id` após switch-context (não header como fonte de verdade) |
| Branch | claim `branch_id` após switch-context |
| Product | `client_id` / audience / scope |
| Roles / Claims / Permissions | resolvidos na emissão; **permissions efetivas no JWT** (ADR-005) |

### Fases de implementação

#### Fase 0 — Fundamentos (código) ✅

- Pacotes NuGet: OpenIddict 7.5.0, Identity EF 10.0.3.
- `IdentityDbContext` + migration Identity/OpenIddict; `User` skeleton.
- Wiring: `AddIdentityFoundation`, AuthN/AuthZ na pipeline, dev signing certs.
- Smoke: `AuthFoundationScenarios` (+ suite Sample intacta).

**Critério de saída:** atendido.

#### Fase 1 — Identidade e Organization/Product scaffold ✅

Aggregates: `Organization`, `Product`, `User`, `Membership`.

- Soft-delete + auditoria (`IdentityAuditableEntity` / campos em `User`).
- CRUD Identity (rotas `api/identity/*`); User via `UserManager` (senha hasheada).
- Seed dev (`IdentitySeed`).
- Testes: create org/product/user, hash password, isolamento membership por org.

**Critério de saída:** atendido.

#### Fase 2 — Authorization Server (OpenIddict) ✅

- Flows: Authorization Code + PKCE; Refresh Token; Client Credentials; grant custom `switch_context`.
- AuthClients/scopes seed (`dev-product-spa`, `dev-product-service`) — Product ≠ AuthClient (D10).
- UI login Razor (`/Account/Login`); consent implícito no seed MVP (D6).
- JWT: claims OIDC + `organization_id`/`branch_id` + **permissions** stub `sso.access` (ADR-005); TTL access 15m.
- `AuthorizationController` (`/connect/authorize|token|userinfo|logout`) + revoke; `TokenClaimsFactory`.
- Testes OIDC: client_credentials, revoke, login, authorize→login, claims factory, switch_context rejeita sem token.

**Critério de saída:** atendido (permissions via factory/switch grant; resolver completo de Role/Permission na Fase 3).

#### Fase 3 — Branch + autorização contextual ✅

- Aggregate `Branch` (`ParentBranchId` estrutural; code único por Organization).
- Catálogo: `Permission`, `Role` (tabela `AuthRoles`), `RolePermission`, `UserRoleAssignment` (Org/Branch?/Product), `ClientProductBinding` (`client_id` → Product).
- Motor `EffectivePermissionsResolver`: Role→Permission; match de Branch **exato** (ADR-004); org-wide (`BranchId` null) soma no branch ativo; sem herança pai→filho.
- switch-context valida Branch ∈ Organization; JWT via `TokenClaimsFactory`.
- Seed matriz HQ vs Filial; testes de não-herança + CRUD Branch.
- ClaimDefinition / UserClaimAssignment adiados (authz primária via Role→Permission no JWT).

**Critério de saída:** atendido.

#### Fase 4 — Credenciais, conta e sessão ✅

- Confirm e-mail (`RequireConfirmedAccount`); forgot/reset password; 2FA TOTP (`LoginWith2fa`, `EnableAuthenticator`).
- `IMailService` (+ `CapturingMailService` nos testes).
- Sessões: revogação em massa via OpenIddict tokens (`IUserSessionService`); logout `/connect/logout` revoga tokens.
- `AuthAuditEvent` + API `GET /api/identity/auth-audit-events`; login/reset/2FA/logout audita.

**Critério de saída:** atendido.

#### Fase 5 — Multi-produto e permissões dinâmicas ✅

- Catálogo `Permission`/`Role`/`RolePermission`/`UserRoleAssignment` (já Fase 3) + teste: código novo entra no JWT na reemissão.
- `perm_ver` dinâmico (stamps de policy) na `TokenClaimsFactory`.
- `MenuItem` por Product + `GET /api/identity/menus/effective` (filtra por permissions).
- Doc: `.ai/CONTEXT/product-integration.md` (+ índice em `integrations.md`).

**Critério de saída:** atendido.

#### Fase 6 — IdPs externos e endurecimento ✅

- Ordem D7: Entra OIDC (homologável) → Google OIDC stub → LDAP stub (`ILdapAuthenticationStub`).
- Catálogo `ExternalIdentityProviders` + `GET /api/identity/external-identity-providers` (só enabled).
- Login: `/Account/ExternalLogin` Challenge/Callback; botões no Login.
- Hardening: CORS allowlist, rate limit (`/Account/*`, `/connect/token|authorize`), lockout config, signing Dev vs Production PFX.
- P-004 Aceito: Production `AutoMigrate=false` por default.
- Docs: `phase6-hardening.md`; checklists security/release atualizados.
- Migration: `Phase6ExternalIdpsHardening`.

**Critério de saída:** Entra pronto para homologação via config; security/release checklists preenchidos.

## Riscos

| Risco | Impacto | Mitigação |
|-------|---------|-----------|
| Escopo monolítico | PRs irrevisáveis | Entregar só por fases |
| Token inchado / permissions stale | Authz frágil / headers grandes | TTL curto, `perm_ver`, revogação, monitorar tamanho (ADR-005) |
| Vazamento cross-org | Segurança | Filtros `OrganizationId` + testes |
| Acoplamento OpenIddict/Identity no Domain | Testabilidade | Ports/adapters |
| Auth parcial | Falsa proteção | AuthN+AuthZ juntos |
| Migrations auto em produção | Outage | P-004 antes de go-live |
| PII em logs | Vazamento | Redação; security checklist |
| Contexto ativo errado (multi-org) | Authz errada | switch-context + claims no token |
