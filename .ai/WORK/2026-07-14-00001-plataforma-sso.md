# Feature Plan — 00001 Plataforma SSO

> Arquivo: `.ai/WORK/2026-07-14-00001-plataforma-sso.md`  
> Template: `.ai/TEMPLATES/feature-plan.md`  
> Status: **Planejamento** (aguardando validação das decisões abertas)  
> Data: 2026-07-14

## Objetivo

Entregar uma plataforma centralizada de **Single Sign-On (SSO)** baseada em **OpenIddict**, com identidade em **ASP.NET Identity**, padrões **OAuth 2.1**, **OpenID Connect** e **JWT**, cobrindo autenticação, autorização e gestão de identidade para todos os produtos do ecossistema.

A solução deve permitir:

- identidade única compartilhada entre produtos;
- autorização híbrida (Roles + Claims + permissões dinâmicas);
- multi-produto, multi-organização (tenant) e multi-filial (branch);
- contexto completo de autorização: **Usuário → Organização → Branch → Produto → Role → Claims → Permissões**.

## Contexto

### Estado atual do repositório

- Solution **CQRS Full (.NET 10)** com Clean Architecture (BAYSOFT.Abstractions + Forge).
- Único aggregate de negócio: **Sample** (scaffold).
- AuthN/AuthZ **não configurados** (`UseAuthentication` comentado; sem Identity/JWT/OpenIddict).
- Persistência: SQL Server + EF Core (`DefaultDbContext`).
- Capacidades reais de SSO: **ainda não implementadas** (ver `CONTEXT/business.md`, `CONTEXT/stack.md`, decisões P-001 / P-006).

## Escopo

### Inclui (visão do produto — feature 00001)

- Cadastro e gerenciamento de usuários
- Autenticação centralizada (OAuth 2.1 / OIDC)
- Gerenciamento de credenciais
- Recuperação e redefinição de senha
- Confirmação de e-mail
- Autenticação em dois fatores (2FA)
- Gerenciamento de sessões
- Emissão e revogação de tokens
- Auditoria de autenticação e autorização
- Roles e Claims globais e por produto
- Permissões customizadas e dinâmicas (sem recompilação das apps cliente)
- Políticas de autorização e herança de permissões
- Multi-produto (clients OAuth, scopes, roles/claims/menus por produto)
- Multi-organization (tenants, isolamento lógico, IdPs externos por org)
- Multi-branch (filiais, vínculo usuário×branch, roles/claims por branch)
- Resolução de autorização no contexto completo da requisição

### Fora de escopo (fase inicial / explícito)

- Migração/remoção do aggregate Sample (pode coexistir até estabilização)
- UI completa de administração (priorizar API + fluxos OIDC; portal admin em fase posterior)
- Implementação completa de cada IdP externo (LDAP, Entra ID, Google) — planejar contratos/adapters; entregar provedores por prioridade
- Microserviços / event bus distribuído (manter monolito modular CQRS até ADR contrário)
- Observabilidade avançada (APM) e CI/CD — dependem de P-002 / P-003
- Compliance formal (SOC2, etc.) além das práticas de segurança do playbook

### Premissa de entrega

A feature **00001** é tratada como **épico**. A implementação ocorre em **fases sequenciais** (ver Abordagem). Cada fase deve ter PR revisável, testes e atualização de `CONTEXT/`.

## Abordagem

### Princípios de arquitetura

1. Preservar **Clean Architecture + CQRS Full** (Sample como espelho de estrutura).
2. Composition root em `SSO.Middleware`; AuthN/AuthZ habilitados de forma completa (não parcial).
3. Domain não depende de OpenIddict/Identity — portas/interfaces no Domain; adapters em Infrastructure.
4. Writer/Reader + `CommitAsync` como unidade de trabalho (sem UoW paralelo).
5. Rotas de gestão: `api/{context}/{resource}`; endpoints de protocolo OIDC seguem convenção OpenIddict (`/connect/*`).
6. Classes `sealed`, Ids `Guid`, localization `pt-BR` (+ `en-US` nos testes).
7. Não inventar regras: decisões abertas bloqueiam a fase correspondente.

### Bounded context proposto

| Opção | Descrição | Recomendação |
|-------|-----------|--------------|
| A | Manter tudo em `Default` / `DefaultDb` | Aceitável no MVP; schema único |
| B | Novo context `Identity` / `IdentityDb` | Isola domínio de identidade do Sample |

**Recomendação de planejamento:** iniciar em **context `Identity`** (Option B) para não misturar scaffold Sample com o núcleo SSO. Confirmar em ADR-006 antes da Fase 1.

### Modelo de domínio (aggregates / entidades raiz previstas)

```text
Identity (context)
├── Organization          # tenant
├── Branch                # filial (hierarquia opcional)
├── Product               # produto do ecossistema
├── ApplicationUser       # extensão Identity (perfil)
├── Membership            # User ↔ Organization [↔ Branch]
├── Role                  # global | por Product | por Branch
├── ClaimDefinition       # claim tipada (global/produto)
├── Permission            # permissão dinâmica
├── RolePermission        # vínculo role → permission (+ herança)
├── UserRoleAssignment    # role no contexto Org/Branch/Product
├── UserClaimAssignment   # claim no contexto Org/Branch/Product
├── ProductEnablement     # Organization → Products habilitados
├── AuthClient            # cliente OAuth (por produto)
├── Scope                 # escopos OIDC/OAuth
├── Session               # sessões ativas (ou view sobre tokens/persisted grants)
├── ExternalIdentityProvider
└── AuthAuditEvent        # auditoria AuthN/AuthZ
```

OpenIddict entities (Applications, Authorizations, Scopes, Tokens) podem ser store próprio OpenIddict mapeado no mesmo DbContext ou schema dedicado — **decisão ADR-001**.

### Contexto de autorização (runtime)

Toda decisão de authz resolve:

| Dimensão | Origem típica |
|----------|----------------|
| User | subject do token (`sub`) |
| Organization | claim/`acr`/header/tenant selector (a definir) |
| Branch | claim ou parâmetro de contexto ativo |
| Product | `client_id` / audience / scope prefix |
| Roles / Claims / Permissions | store + política efetiva (com herança) |

Token JWT deve carregar **contexto mínimo necessário** (evitar token inchado); detalhes granulares via endpoint de introspecção / userinfo / API de autorização — **ADR-005**.

### Fases de implementação

#### Fase 0 — Fundamentos e ADRs (sem feature de negócio completa)

- Registrar decisões (OpenIddict, Identity, tenant, context DB).
- Atualizar `CONTEXT/business.md`, `glossary.md`, `Decisions.md` com vocabulário SSO.
- Pacotes NuGet: OpenIddict, Identity, EF stores.
- Habilitar pipeline Auth de forma segura (dev certificates / signing keys).

**Critério de saída:** ADRs aceitos; Dil acoplado; app sobe com schemes configurados.

#### Fase 1 — Identidade e Organization/Product scaffold

Aggregates mínimos: `Organization`, `Product`, `ApplicationUser` (+ Identity tables), `Membership` (user↔org).

- Commands/Queries CRUD alinhados ao padrão Sample.
- Migration inicial Identity + domínio.
- Seed de organização/produto de desenvolvimento.

**Critério de saída:** criar org/produto/usuário via API; senha hasheada; isolation por Organization nas queries.

#### Fase 2 — Authorization Server (OpenIddict)

- Flows prioritários: Authorization Code + PKCE; Refresh Token; Client Credentials (M2M).
- Clients e Scopes por Product.
- Endpoints discovery, token, authorize, userinfo, revocation, introspection (conforme necessidade).
- Emissão JWT com claims padrão OIDC + claims de contexto (org/product) mínimos.

**Critério de saída:** app cliente de exemplo obtém e renova access token; revoke funciona.

#### Fase 3 — Branch + autorização contextual

- Aggregate `Branch` (hierarquia opcional).
- Assignments de Role/Claim/Permission por Org/Branch/Product.
- Motor de resolução de permissões efetivas + herança.
- Políticas ASP.NET (`IAuthorizationHandler` / requirement) consumindo o contexto.

**Critério de saída:** mesmo usuário com perfis distintos por Branch/Product; testes de matriz de autorização.

#### Fase 4 — Credenciais, conta e sessão

- Confirmação de e-mail; forgot/reset password (ativar `IMailService` real ou adapter).
- 2FA (TOTP no mínimo).
- Gestão de sessões / logout / revogação em massa.
- Auditoria de login, falhas, emissão/revogação, mudanças de role.

**Critério de saída:** fluxos de conta cobertos por testes de integração; audit trail consultável.

#### Fase 5 — Multi-produto operacional e permissões dinâmicas

- Catálogo de Permissions sem redeploy das apps cliente (versão/etag de policy opcional).
- Menus/funcionalidades como recursos autorizáveis (contrato de API para clients).
- Claims/roles globais vs por produto estabilizados.
- Documentação de integração para times de produto.

**Critério de saída:** novo Permission/Role aplicável sem recompilar clients que consultam a API de autorização.

#### Fase 6 — IdPs externos e endurecimento

- Contratos `ExternalIdentityProvider` por Organization.
- Provedores prioritários (ordem a definir): Microsoft Entra ID, Google, LDAP.
- Hardening: rate limit login, lockout, key rotation, CORS/origins, secrets.
- Estratégia produção: migrations fora do auto-apply se P-004 assim decidir.

**Critério de saída:** pelo menos um IdP externo em ambiente de homologação; checklist security/release ok.

## Riscos

| Risco | Impacto | Mitigação |
|-------|---------|-----------|
| Escopo monolítico da 00001 | Atraso / PRs irrevisáveis | Entregar só por fases; cada fase com DoD |
| Token JWT com permissões excessivas | Tokens grandes / stale permissions | Claims mínimas + endpoint de autorização / versionamento de policy (ADR-005) |
| Isolamento multi-tenant frágil | Vazamento cross-org | Specs + filtros obrigatórios por `OrganizationId` em Reader; testes de isolamento |
| OpenIddict + Identity + CQRS mal acoplados | Domain sujo / difícil teste | Ports no Domain; adapters em Infrastructure; Domain Services só para regras de negócio próprias |
| Pipeline Auth parcial | Falsa sensação de segurança | Habilitar AuthN+AuthZ juntos; `[Authorize]` em APIs de gestão |
| Migrations auto no startup em produção | Drift / outage | Resolver P-004 antes de go-live |
| PII em Notifications (JSON logs) | Vazamento de dados | Redação/pátios de log; checklist security |
| Herança de permissões ambígua | Authz inconsistente | Documentar algoritmo em `business.md` + testes de matriz |
| Concurrent membership multi-org | Contexto ativo errado | Contrato explícito de “active organization/branch” no token ou sessão |
| IdPs externos heterogêneos | Escopo infinito na fase 6 | Um provedor por incremento; interface estável |

## Glossário proposto (para `CONTEXT/glossary.md` após validação)

| Termo | Significado |
|-------|-------------|
| **Organization** | Tenant; isolamento lógico de identidade e authz |
| **Branch** | Filial / unidade dentro de uma Organization |
| **Product** | Sistema do ecossistema que consome o SSO |
| **AuthClient** | Cliente OAuth/OIDC registrado (ligado a um Product) |
| **Membership** | Vínculo usuário ↔ organização (e opcionalmente branch) |
| **Permission** | Capacidade autorizável dinâmica |
| **Effective permissions** | Conjunto resolvido no contexto User×Org×Branch×Product |
| **Authorization Server** | Este serviço (OpenIddict) emissor de tokens |
