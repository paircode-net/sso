# Feature Plan — 00008 Claims tipadas (ClaimDefinition / UserClaimAssignment)

> Arquivo: `.ai/WORK/2026-07-16-00008-claims-tipadas.md`  
> Template: `.ai/TEMPLATES/feature-plan.md`  
> Status: **Implementado** (D-00008-1..5 aceitas)  
> Data: 2026-07-16  
> Depende de: Fase 3 motor authz (Role→Permission); ADR-004/005  
> Nota: adiado explicitamente na Fase 3 do épico 00001

## Objetivo

Completar o modelo de autorização híbrido com **claims tipadas de domínio** (catálogo `ClaimDefinition` + atribuições `UserClaimAssignment` no contexto Org/Branch/Product), emitindo claims adicionais no JWT quando fizer sentido — sem substituir permissions como mecanismo primário de gate.

## Contexto

- AuthZ primária atual: Role → Permission → claim `permissions` no JWT.
- Plano 00001 previa ClaimDefinition / UserClaimAssignment; adiados de propósito.
- Casos de uso: atributos (`department=finance`), flags tipadas (`can_export=true`), IDs de negócio que não são permissions.

## Escopo

### Inclui

**Catálogo**

- `ClaimDefinition`: code, value type (string/int/bool/json), scope (global/product), description, enabled.
- CRUD `api/identity/claim-definitions` (platform admin).

**Atribuição**

- `UserClaimAssignment`: User + ClaimDefinition + value + Organization + Branch? + Product?.
- Match de Branch: **exato** (mesmo ADR-004) no MVP desta feature; herança fica em 00009.
- CRUD assignments (org admin no tenant; platform global se aplicável).

**Emissão no token**

- `TokenClaimsFactory` / resolver: além de `permissions`, emitir claims tipadas efetivas do contexto.
- Convenção de nomes: prefixo `sso_c_` ou claim type = code do catálogo (decidir para não colidir com OIDC std).
- `perm_ver` **ou** novo `claim_ver` quando assignments mudam (preferir estender policy version provider).

**Docs**

- Quando usar Permission vs Claim (guia curto em product-integration).
- Products **não** devem usar claims tipadas no lugar de permissions para authz de rota — claims são atributos.

### Fora de escopo

- Claims OIDC std (`email`, `name`) — já Identity/OpenIddict.
- Transformação arbitrária de claims de IdP externo (mapeamento IdP → ClaimDefinition pode ser fase 2 desta feature).
- Policy engine ABAC completo (só emissão de atributos).

## Abordagem

### Fase A — Modelo + API

1. Aggregates Domain + EF configs + migration.
2. CQRS CRUD + validators (unicidade code; value conforme type).
3. Seed exemplo: `department` string, `mfa_required` bool.

### Fase B — Resolução + JWT

1. `EffectiveClaimsResolver` paralelo ao de permissions (mesmas regras de contexto).
2. Integração `TokenClaimsFactory` em login/refresh/switch_context.
3. Testes: claim aparece/desaparece com assignment e contexto.

### Fase C — Contrato

1. Atualizar `SsoClaimTypes` / doc.
2. SDK (00004) helper `GetSsoClaim<T>("department")` opcional.

## Critérios de aceite

- [x] Admin cria ClaimDefinition e atribui valor a usuário no contexto org/branch/product.
- [x] Novo access token contém a claim; contexto errado não contém.
- [x] Mudança de assignment altera version stamp (`claim_ver`).
- [x] Documentação deixa claro: gates de rota = `permissions`; claims = atributos.
- [x] Tamanho do JWT monitorado (teste com N=50 claims).

## Riscos

| Risco | Mitigação |
|-------|-----------|
| Token inchado | Limite de claims por token; só emitir no audience do product; TTL curto |
| Colisão com claims OIDC | Namespace/prefixo obrigatório |
| Products usarem claim como authz frágil | Docs + exemplos SDK só com RequirePermission |
| Duplicação Role vs Claim | Guia de modelagem no CONTEXT |

## Decisões aceitas

- [x] **D-00008-1:** Claim no JWT com prefixo **`sso_c_{code}`** (ex.: `sso_c_department`). Evita colisão com OIDC std e claims SSO; segue estilo legível do `SsoClaimTypes`.
- [x] **D-00008-2:** Criar claim **`claim_ver`** separado do `perm_ver`. Version stamp próprio para mudanças de `ClaimDefinition` / atribuições; produtos passam a ler os dois etags.
- [x] **D-00008-3:** Origem **`UserClaimAssignment` + `RoleClaim`**. Atributos podem vir do papel (herdados) e/ou por usuário no contexto org/branch/product.
- [x] **D-00008-4:** Value types = **escalares** (`string`, `int`, `bool`). Sem JSON/datetime no MVP (token enxuto, validação simples, `GetTypedClaim<T>` direto).
- [x] **D-00008-5:** Precedência **`UserClaimAssignment` vence `RoleClaim`** para o mesmo `ClaimDefinition` no mesmo contexto (override individual determinístico).

### Impactos das decisões no escopo

- **D2** exige estender o motor de version stamp com um provider próprio (`claim_ver`) ou um segundo cálculo no `PermissionPolicyVersionProvider`; atualizar `SsoClaimTypes` e o contrato em `product-integration.md` (produtos leem `perm_ver` **e** `claim_ver`).
- **D3** adiciona aggregate/EF config **`RoleClaim`** (Role + ClaimDefinition + value) além de `UserClaimAssignment`, e o `EffectiveClaimsResolver` faz o merge.
- **D5** define a regra de merge do resolver: coletar RoleClaims efetivas → sobrepor com UserClaimAssignments do mesmo `code`/contexto.
